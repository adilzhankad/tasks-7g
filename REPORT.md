# Отчёт по Задаче 1 — Анализатор энтропии сетевых пакетов

## Что нужно было сделать

Нужно было написать программу под Linux, которая:
1. Слушает сетевой интерфейс и читает пакеты
2. Фильтрует их по IP-адресу
3. Считает энтропию (насколько разнообразный трафик) по скользящему окну
4. Периодически выводит результат в консоль

## Почему я выбрал Go и libpcap

Я решил писать на **Go**, потому что у него есть горутины — это удобно, когда нужно одновременно и пакеты ловить, и результат выводить. Плюс в Go есть отличная библиотека `gopacket`, которая оборачивает libpcap и делает работу с пакетами простой.

DPDK я не стал использовать — он мощный, но требует кучу настроек (hugepages, специальные драйверы). Для этой задачи это перебор, libpcap справляется отлично.

## Как я решал задачу

### Шаг 1 — Подготовил окружение

Установил Go и libpcap-dev:
```bash
sudo apt-get install -y golang-go libpcap-dev
```

Создал Go-модуль:
```bash
go mod init github.com/adilzhankad/tasks-7g
```

### Шаг 2 — Написал программу

Код разбит на 4 файла, каждый отвечает за свою часть:

- `main.go` — точка входа, парсинг флагов, основной цикл вывода
- `capture.go` — открытие интерфейса, BPF-фильтр, горутина захвата пакетов
- `window.go` — скользящее окно (структура PacketInfo, добавление, снимок) + mutex
- `entropy.go` — расчёт энтропии Шеннона

Расскажу по частям, что там происходит.

**Аргументы командной строки.** Через флаги передаю интерфейс, IP, размер окна и интервал вывода:
```go
iface := flag.String("i", "", "Network interface")
hostIP := flag.String("host", "", "IP address to filter")
windowSize := flag.Int("window", 100, "Sliding window size")
interval := flag.Duration("interval", 5*time.Second, "Print interval")
```

**Захват пакетов.** Открываю интерфейс через pcap и ставлю BPF-фильтр. BPF — это фильтр на уровне ядра, то есть ненужные пакеты даже не доходят до программы. Это быстро и эффективно:
```go
handle, err := pcap.OpenLive(*iface, 65535, true, pcap.BlockForever)
handle.SetBPFFilter(fmt.Sprintf("host %s", *hostIP))
```

**Скользящее окно.** Я храню размеры последних N пакетов в слайсе. Когда окно заполняется — удаляю самый старый элемент:
```go
if len(window) >= *windowSize {
    window = window[1:]
}
window = append(window, size)
```

**Расчёт энтропии.** Использую формулу Шеннона: `H = -Σ p(x) * log2(p(x))`. Смотрю, какие размеры пакетов встречаются в окне, считаю частоту каждого и вычисляю энтропию. Чем разнообразнее размеры — тем выше число.

**Два параллельных процесса.** В отдельной горутине ловлю пакеты и кладу в окно. В основном потоке — таймер, который каждые N секунд считает энтропию и выводит. Между ними — mutex, чтобы не было гонки при доступе к окну.

### Шаг 3 — Собрал

```bash
go build -o entropy-sniffer .
```

Скомпилировалось без ошибок.

### Шаг 4 — Протестировал

Запустил программу:
```bash
sudo ./entropy-sniffer -i enp4s0 -host 192.168.10.3 -window 50 -interval 3s
```

Параллельно гонял трафик — пинги, curl на разные сайты. Программа ловила пакеты и показывала энтропию каждые 3 секунды. Видно было, как при однотипном трафике (только ping) энтропия низкая (~1.0), а при смешанном (сайты + DNS + ping) поднималась до 3.5.

## Какие решения я принял и почему

- **Размер пакета как символ для энтропии** — это самая естественная характеристика пакета. Если все пакеты одного размера — трафик однородный, энтропия низкая. Разные размеры — разнообразный трафик, энтропия высокая.

- **Mutex вместо каналов** — у меня простой случай: одна горутина пишет в окно, другая читает. Mutex тут проще и понятнее, чем городить каналы.

- **BPF-фильтр** — фильтрация прямо в ядре, а не в коде. Программа не тратит время на пакеты, которые ей не нужны.

- **Graceful shutdown** — обрабатываю Ctrl+C (SIGINT), чтобы программа корректно закрывала pcap handle, а не падала.

## Итог

Написал CLI-программу на Go, которая делает всё, что требовалось: ловит пакеты, фильтрует по IP, считает энтропию в скользящем окне и выводит её с заданным интервалом. Протестировал на реальном трафике — работает.
